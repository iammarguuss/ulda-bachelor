# Performance Profiling and Optimization

## Scope and target selection

Lab 8 was implemented primarily for:

- `applications/max_speed_test_server`

Reason:

- it is already the most performance-oriented application in the repository
- it is designed for in-memory HTTP stress experiments
- it avoids MySQL/network-storage effects that would blur CPU and allocation analysis

`applications/ulda-crud` was not used as the main profiling target because its results would be dominated by database behavior and connection variability rather than by the ULDA HTTP hot path itself.

## Chosen profiling tools

The following practical tools were used:

- Node.js built-in CPU profiling with `--cpu-prof`
- high-resolution timing with `process.hrtime.bigint()`
- in-process profiling counters exposed through `/stats`
- memory snapshots via `process.memoryUsage()`
- repeatable benchmark runner in `applications/max_speed_test_server/scripts/perf-scenarios.js`

Why this combination was chosen:

- it stays lightweight and reproducible
- it matches the actual server workload instead of synthetic isolated functions
- it allows both end-to-end metrics and narrower hot-path timing

## Selected metrics

The report tracks:

- average response latency
- `p95` latency
- throughput in requests per second
- per-method average latency
- heap delta during each scenario
- cumulative hot-path timings for `verify`, `parseBytes`, `encodeBytes`, `storeRead`, `storeWrite`

These metrics matter because `max_speed_test_server` is intended for high-frequency request experiments where both latency and throughput are relevant, and the data path is small enough that repeated parsing/encoding/allocation overhead becomes visible.

## Scenarios and datasets

The same CRUD-style lifecycle was used in all scenarios:

- `POST /records`
- repeated `PUT /records/:id`
- two `GET /records/:id` reads during the lifecycle
- `DELETE /records/:id`

Scenario definitions:

| Scenario | Clients | Updates/client | Content bytes |
| --- | ---: | ---: | ---: |
| small | 10 | 1 | 32 |
| medium | 25 | 3 | 256 |
| large | 50 | 5 | 2048 |

The payloads are deterministic and generated by the benchmark script, so runs are repeatable within the limits of the local machine.

### Baseline hot spots

The built-in CPU profile produced a weak signal for this short-lived in-memory server and mostly highlighted startup/framework frames. Because of that, the main hot-spot ranking is based on the in-server high-resolution counters from the large scenario.

## Identified hot spots

Three real hot spots were selected from the baseline evidence:

1. `verify` dominated cumulative server time in all scenarios, especially large.
2. `parseBytes` was the most expensive repeated non-ULDA utility on the input path.
3. `encodeBytes` was the main repeated output-side utility, especially for repeated reads.

Additional lower-tier hot spot:

- `storeWrite` showed repeated allocation/update work in the write path.

## Implemented optimizations

The optimization work stayed narrowly focused on the speed-test server and did not redesign the whole repository.

### 1. Faster input parsing

`parseBytes` was optimized by:

- returning `Buffer` inputs directly
- using a zero-copy `Buffer.from(buffer, offset, length)` path for `Uint8Array`
- avoiding unconditional `trim()` when the input string has no surrounding spaces
- using common-case fast paths for `hex`, `base64`, and default formats before falling back to full normalization

### 2. Read-path encoding cache

The server now stores:

- cached `ulda_key_b64`
- optionally cached `content_b64` for payloads up to `512` bytes

This reduces repeated `encodeBytes()` work for the common base64 read path without caching arbitrarily large payloads.

### 3. Verification-path preparation cache

Each stored record now keeps a prepared parsed-signature form:

- `parsedKey`

This allows the speed-test server to reuse already-imported ULDA signature metadata when checking the next state transition.

### 4. Write-path allocation cleanup

The old `updatedAt` field was removed from the in-memory records because it was not used by any endpoint in this application. This reduces unnecessary write-path allocation work in the benchmark target.

## Before/after comparison

Improvement formula used:

```text
improvement % = (before - after) / before * 100
```

For throughput, where bigger is better:

```text
improvement % = (after - before) / before * 100
```

### Per-scenario comparison

| Scenario | Avg latency improvement | p95 improvement | Throughput improvement | Heap delta improvement |
| --- | ---: | ---: | ---: | ---: |
| small | +2.65% | +8.61% | +5.51% | +1.04% |
| medium | +8.75% | +6.09% | +22.22% | -3.55% |
| large | -29.18% | -4.71% | -2.39% | +7.91% |

### Aggregate comparison across all scenarios

Using all requests from all three scenarios together:

- baseline aggregate throughput: `539.45 req/s`
- post-optimization aggregate throughput: `564.38 req/s`
- aggregate throughput improvement: `+4.62%`
- baseline weighted average latency: `27.678 ms`
- post-optimization weighted average latency: `33.245 ms`
- weighted average latency change: `-20.11%` improvement formula result, meaning a regression in latency

## New hot spots after optimization

After optimization:

- `verify` remained the dominant hot spot and became even more dominant in the large scenario
- `encodeBytes` pressure dropped sharply because repeated base64 encoding was reduced
- `storeWrite` became more visible because the server now maintains small caches for frequently read values

The supplementary post-optimization CPU profile also showed visible time in Express/query-string stack frames, which suggests framework overhead becomes more visible once some application-side utility work is reduced.

## Interpretation

The optimization pass produced mixed but still useful results:

- small and medium scenarios improved in both throughput and latency
- large-scenario heap growth improved modestly
- large-scenario latency and throughput regressed, which means the current caching strategy does not scale linearly for bigger payloads

So the final academically defensible conclusion is:

- the profiling methodology identified real hot paths
- targeted optimizations improved some scenarios materially
- the large scenario still exposes unresolved trade-offs, especially around ULDA verification cost and cache/write-path pressure

## Limitations

- measurements were taken on a local Windows machine, so absolute numbers are environment-specific
- Node.js built-in CPU profiles were less informative than expected for this short-lived in-memory workload
- the benchmark is repeatable, but still subject to local scheduler and JIT variability
- the current report focuses on server behavior, not browser-side benchmark pages
- no claim is made that the final version is globally “faster” under every workload; only the measured scenarios above are claimed
